# Producer Chat Integration Design Spec

## Overview
The goal of this feature is to add a persistent chat area to the Lobster Radio web interface, allowing the user to chat with a "Producer" agent. This agent acts as a backstage intermediary: it chats with the user, secretly extracts preferences (like mood changes, likes/dislikes, or skip requests), and updates the Memory System so the DJ (`NarratorAgent`) can naturally incorporate these insights into its on-air commentary.

## Requirements
1. **Persistent Chat Interface**: Replace the current right sidebar (History/Queue) with a persistent chat window.
2. **Producer Agent**: A new LLM agent that chats with the user and outputs both a natural language reply and a structured JSON payload containing extracted directives.
3. **Implicit System Updates**: The system must use the Producer's structured output to update user preferences, the Memory System, and potentially trigger UI actions (like skipping a track) without requiring manual buttons.
4. **DJ Integration**: The extracted insights (`memoryInsight`) must flow into the DJ's memory so the next narration reflects the user's chat requests.

## Architecture & Implementation

### 1. Frontend Modifications
- **Layout Change**: Update `frontend/src/pages/HomePage.tsx`. The right grid column will now be dedicated to a new `ChatPanel` component.
- **Relocating Old Components**: Move `TrackQueue` and `HistoryPanel` to a secondary location (e.g., underneath the player or in a collapsible section) so they are still accessible but don't compete with the chat for space.
- **ChatPanel Component**: A standard chat interface with a message list and input box. It will call a new backend endpoint (`/api/chat`).
- **Client-Side Triggers**: If the `/api/chat` response indicates `skipRequested: true`, the frontend should automatically invoke the `onNextTrack` handler.

### 2. Producer Agent (`agents/src/agents/ProducerAgent.ts`)
- **Responsibility**: Act as the "behind-the-scenes" radio producer.
- **Input**: User's message, chat history, current track info, and current user preferences.
- **Output Schema**:
  A structured output (e.g., using tool calling or forced JSON schema in Claude) with two parts:
  ```typescript
  {
    reply: string; // The natural language response to the user
    directives: {
      suggestedMood?: string; // If the user implies a mood shift
      likedArtist?: string;
      dislikedArtist?: string;
      skipRequested?: boolean; // True if the user asks to change the song now
      memoryInsight?: string; // A short note for the DJ (e.g., "Listener wants upbeat pop next")
    }
  }
  ```

### 3. Backend Integration (`backend/src/api/...` or similar route handler)
- Create an endpoint (e.g., POST `/api/chat`) to handle messages from the frontend.
- **Workflow**:
  1. Forward message to `ProducerAgent`.
  2. Parse the response.
  3. If `directives.suggestedMood` exists, update the user's current mood state.
  4. If `directives.memoryInsight` exists, append it to the user's `Preferences` object (or the new Memory System) so the `NarratorAgent` sees it on the next track.
  5. If `directives.likedArtist`/`dislikedArtist` exists, log them to history/preferences.
  6. Return the `reply` and `skipRequested` flag to the frontend.

### 4. DJ Narration Integration
- Update the `NarratorAgent` prompt or its data pipeline to ensure any `memoryInsight` generated by the Producer is fed into the DJ's context.
- Example DJ prompt injection: *"Producer note: The listener mentioned they are feeling tired and want something energetic. Acknowledge this."*

## Error Handling & Edge Cases
- **LLM Failures**: If the Producer agent fails to return valid JSON, gracefully fallback to a generic reply ("Got it, I'll let the DJ know.") and empty directives.
- **Concurrent Requests**: Ensure that changing the mood via chat doesn't conflict with manual mood dropdown changes if they happen simultaneously.

## Testing
- Unit tests for `ProducerAgent.ts` to ensure it accurately extracts directives from sample user messages (e.g., "skip this" -> `skipRequested: true`).
- UI testing to ensure the chat panel scrolls correctly and that `skipRequested` successfully triggers a track change in the frontend.
